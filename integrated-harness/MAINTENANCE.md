# 編排規則維護說明

只有人類擁有者可以修改本文件及 `ORCHESTRATOR.md` 的授權邊界。計畫閘門、核准
模式與使用流程以 `ORCHESTRATOR.md`、政策檔及 README 為準；本文件只保留決策理由、
修改路由與已知限制。

## 各章設計理由

| 章節 | 防範問題 | 被否決的替代方案 |
|---|---|---|
| A 任務分解 | 重疊修改、依賴遺漏 | 一律最大化平行派工；協調成本過高 |
| B 模型路由 | 能力與成本錯配 | 固定所有工作使用同一模型 |
| C 任務委派 | 無狀態代理自行補完 | 直接轉貼整段對話；脈絡過多且可能洩密 |
| D 驗收重試 | 以主觀印象宣告成功 | 無限重試或失敗後降低標準 |
| E 自主權 | 未授權外部副作用 | 只依工具權限判斷授權 |
| F 失敗回報 | 隱藏未完成與風險 | 只回報最後錯誤訊息 |
| G 資源意識 | Context 與成本耗盡 | 所有細節都由編排層親自處理 |
| H 計畫閘門 | 施作早於結構化拆解 | 只靠 prompt 承諾先計畫；無法確定檢查 |
| I 反過度設計 | 範圍與架構膨脹 | 以最佳實務作為無條件擴張理由 |

## 何時修改

- 更換平台或新增、移除模型：更新政策檔的「模型對應」；B 章的層級定義
  只在職責劃分本身改變時才修改。
- 技術堆疊或單位情境改變：只更新政策檔的「單位與技術堆疊」；正式環境授權邊界
  改變時再同步 E 章、政策檔與相關 hook。
- 專案風險等級改變：由人類在 `strict`、`standard` 與 `light` 之間調整政策；
  涉及正式環境、個資或共享基礎設施的專案應維持 `strict`。
- 新的失敗模式實際發生：先補可重現測試，再修改最接近的單一章節。
- 驗收流程改變：更新 D；不要藉機擴張 I 章允許的工作範圍。
- Hook 規則改變：hook 是確定性規則的單一事實來源；正文只引用 hook 的責任，
  不複製 regex 或實作細節。
- `harness/` 與 `integrated-harness/` 是能力不同的產品，不要求 hooks 逐行同步。共同
  失敗模式修補後必須評估另一產品是否受影響，只有行為契約相同時才移植並各自驗證；
  移植後把攻擊樣本加入根層 `tests/claude_hook_parity_test.sh` 的共同行為語料，
  由該測試守護兩產品對同一輸入的判定一致。

## 已知限制

- 個資防線分兩層、能力不對稱：`block_pii_prompt.py`（UserPromptSubmit）只能整段
  阻擋，因 Claude Code 的 UserPromptSubmit 不支援改寫提示內容；
  `redact_sensitive_info.py`（PreToolUse）才支援「去識別化後放行」，因
  PreToolUse 有 `updatedInput` 可改寫 `tool_input`。使用者若在 prompt 階段被擋下，
  仍需自行遮蔽後重送，非本工具自動處理。
- 個資偵測規則（台灣身分證字號、手機號碼、Email）為初版樣式，不含學號、地址、
  護照號碼等其他個資類型；如需擴充，於 `redact_sensitive_info.py` 的 `RULES`
  新增規則即可（`block_pii_prompt.py` 直接複用同一份定義，不需重複維護）。
- 尚未實作攔截已知正式環境主機或部署命令的 hook；環境命名不一致可能誤攔。
- 尚未實作計畫關卡前攔截非 Bash／檔案工具外部副作用的 hook；matcher 依新增工具決定，
  且可能攔截唯讀 API。

- 規則無法判斷計畫內容是否誠實完整；`strict` 仍需人類實際審查，
  `standard`／`light` 的擁有者也必須把計畫品質當作已知風險。
- `strict` 的吞吐量上限受人類審批速度限制。若人類因審批疲勞而不看就核准，
  防線會名存實亡；規則本身無法防止。
- I 章與 D 章存在天然張力：驗收標準越完備，最簡解法通常越大。控制過度設計的
  真正槓桿是人類精確定義驗收標準。
- 允許修改範圍只在 `strict`／`standard` 下檢查檔案工具的目標路徑；無法理解
  Bash、間接執行或未來工具的語意，不能取代 permissions、sandbox 與人工審查。
- Shell token 判斷與 regex hooks 不是完整 shell parser，無法涵蓋混淆、別名、間接執行
  或所有未來工具；安全 hooks 只是縱深防禦，需搭配 Secret Manager、SAST、
  CI secret scanner、程式碼審查與人工抽查。
- `strict` 核准紀錄是 60 分鐘的專案層級時間窗，並綁定拆解文件 SHA-256；
  同一份計畫在有效期間內的所有寫入均視為同一核准範圍，計畫邊界必須足夠明確。
- `standard`／`light` 沒有人類授權環節；`standard` 只增加檔案工具的範圍限制，
  `light` 只構成思考紀律，兩者都不可解讀為操作已獲人類授權。

## 維護驗證

任何修改至少執行：

```bash
bash tests/smoke_test.sh
bash tests/orchestration_test.sh
```

新增規則時先回答：「這條規則防範的錯誤是否已實際發生，或有具體且高可能性的
情境？」若否，不加入正文；記錄為候選即可。
